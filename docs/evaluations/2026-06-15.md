gatecrate の二階ループ（ハーネス自己評価）を localmd に初導入し、実 CI 履歴で評価した初回レポート。
方法論: gatecrate `docs/harness-roi-evaluation.md`。根拠は全て実測（`collect-gate-history.sh`（`gh`）/
`probe-gate-liveness.sh` / `check-posix-portability.sh` / `git log`）。憶測なし。

**導入物**: `scripts/{probe-gate-liveness,collect-gate-history,check-test-compiles,check-posix-portability}.sh`
（kit と byte 一致で vendoring）、`.takt/`（harness-evaluate-cycle / liveness-converge + persona）、
`harness.config.sh` に `PROBE_GATES`。なお `sync-manifest.yaml` の `consumed_scripts` には**まだ加えない**——
これらは gatecrate v0.8.0（現ピン版）に未存在で、`check-kit-drift.sh` が byte 比較で落ちるため。これらを含む
gatecrate 版がリリースされ `harness_kit_version` を上げた時点で consumed_scripts に追加する。

## 生存証明（最優先）

`probe-gate-liveness.sh`: 予防ゲート **title / secrets ともに ALIVE**（合成違反を拒否）。黙って壊れたゲートなし。

## 重要な検出 ①（移植性・即時に実バグ発見）

導入した `check-posix-portability.sh` が localmd の harness に**実害を即検出**:

| 種別 | スクリプト | 問題 |
|---|---|---|
| Termux 専用 shebang | `version-code-bump.sh` / `visual-regression-check-limited.sh` / `test.sh` | `#!/data/data/com.termux/files/usr/bin/sh` ＝**非 Termux 環境（CI・macOS・Git Bash）で起動不能** |
| bashism | `smoke-render-assert-l5.sh`（L47,60,80,81） | `local`（SC3043）＝POSIX sh 未定義 |

**なぜ問題か→だから何をするか**: 長期開発で実行環境が変われば（CI ランナー更新・別 OS・別シェル）これらは黙って壊れる。
→ **strengthen**: shebang を `#!/bin/sh` に統一、`local` を排除。本ゲートを CI に常設すれば再混入を機械防止できる。

## Axis 1（撤去・分母＝CIコスト×頻度）

実 CI 20 runs（`ci.yml`）の抜粋。全衛生ゲートは低コスト、ビルド/テストがコストを支配。

| 層 | 型 | runs | fires | CI秒(total/avg) | 判定 |
|---|---|---|---|---|---|
| **Check documentation currency** | 検出 | 11 | **2** (0.18) | 0.0 | **keep**（実発火＝ドリフトを実際に捕捉している） |
| Build Gradle debug+release bundle | ビルド | 20 | 0 | 1782 / 89.1 | keep（最大コストだが固有・出荷物） |
| Run tests and debug build | テスト | 20 | 0 | 431 / 21.6 | keep（固有） |
| Build Pro preview debug APK | ビルド | 20 | 0 | 548 / 27.4 | keep（固有） |
| Enforce per-file line count limit | 予防 | 20 | 0 | 10 / 0.5 | keep（安価・ALIVE） |
| Enforce harness hard constraints | 予防 | 20 | 0 | 0 | keep（安価） |
| Scan for committed secrets | 予防 | 20 | 0 | 2 / 0.1 | keep（安全制約・ALIVE） |

**結論: removal-candidate = 0。** removal は3条件論理積（高コスト∧独自性ゼロ∧発火ゼロ）。コストを支配する
ビルド/テストは独自（②で残る）、衛生ゲートは安価（①で残る）。doc-currency は**実際に2回発火**＝価値が実証されている。
gatecrate 自己評価・初代 localmd ROI と同じく、**コスト1軸では何も剪定できない**。

## Axis 2（統合・分母＝維持負荷）— ここが localmd の核心

| シグナル | 実測 | 評価 |
|---|---|---|
| doc churn | docs/ を触ったコミット **311** | **非常に高い** |
| ワークフロー数 | **13** | 高い（認知負荷） |
| scripts 数 | **63**（導入前59） | 高い |
| harness 関連 doc | `docs/harness*` **25本** | **高い**（ハーネスを評価する文書がハーネスを評価する自己言及階層） |

**consolidate-candidate（複数）**:
- **なぜ問題か**: 13 ワークフロー・25 harness doc・311 doc churn は、CI 秒に出ない**継続的な人的維持負荷**。
  ハーネスが「第二のプロダクト」化し、本体（リーダーアプリ）への投資を圧迫する——localmd で実際に観測された症状。
- **だから何をするか（撤去でなく統合/自動化）**: ① emulator/visual 系ワークフロー群の集約、② 25 harness doc を
  単一インデックス配下へ整理、③ doc-currency 系ゲートの重複統合。**提案のみ・人間が承認**。

**過大判定ガード**: doc churn 311 は「無駄」でなく活発な開発の証でもある。重複・手作業規律依存の層だけを統合候補とし、
安価で独自な衛生ゲート（line-limit / secrets / hard-constraints）は明示 keep。

## 二階ループ導入の検証結論（長期開発で使えるか）

| 観点 | 結果 |
|---|---|
| 導入（vendoring） | ✅ kit と byte 一致でコピー・既存 CI に非破壊 |
| 実行 | ✅ probe ALIVE・collect が実 CI データ取得・portability 実行 |
| **実価値（即・実バグ）** | ✅ Termux shebang 3本 + bashism を**導入直後に検出** |
| **実価値（ROI）** | ✅ removal=0／axis-2 で localmd の「重い harness」問題を**定量再現** |

→ **長期開発で使える**: 導入は非破壊・低コストで、実プロジェクトの**現実の移植性バグ**と**維持負荷の核心**を
即座に根拠つきで可視化した。次は consolidate 提案の人間承認と、portability ゲートの CI 常設。

## Phase 5 — 提案（人間が承認・実行）

- **consolidate**: workflow 群・harness doc・doc-currency の統合（提案のみ）。
- **removal**: なし。安全制約はメトリクスで自動撤去しない。
