# ADR-0009: Android Viewより上位で操作状態を管理する

状態: 採用

## 決定事項

ユーザーから見える操作状態と遷移は、Android View層より上位のalways-validなモデル型で表現する。
Viewはモデル状態を描画し、ユーザーのコマンドを中継する。visibility、アニメーション進捗、表示文字列、
nullableなWidgetフィールドなどのViewプロパティを、仕様上の正しい状態として扱わない。

まずナビゲーションメニューを`Closed`または`Open`、Markdownライブラリを`Unselected`、
`Expanded`、`Collapsed`として表現する。選択済みのライブラリ状態は必ず有効な現在地、全一覧、
絞り込み条件を持ち、そこから表示一覧を導出する。開く、閉じる、開閉を切り替える、絞り込む操作は、
有効なモデル入力に対する全域関数とする。

## 検討した選択肢

- Android Viewのプロパティから操作状態を推測する。
- presentationクラスにbooleanとnullableな内容を保持する。
- データだけをモデル化し、ユーザー操作の振る舞いはViewに残す。
- データ上の区別とユーザー操作の遷移をともにViewより上位でモデル化する。

## 選択理由

Viewプロパティは一時点の描画を表すものであり、ユーザー操作の有効な状態ではない。それを仕様として
扱ったため、UI構造の変更時にメニューのスクロールやライブラリの開閉といった振る舞いが失われた。
明示的なvariantは不可能な組み合わせを排除し、再描画をまたいでデータを維持し、すべての有効な操作から
有効な次状態を返せる。また、操作追跡ハーネスがモデル遷移、アダプタ、動作テストを結ぶ基点になる。

## 選択しなかった理由

visibilityとアニメーション値は遷移中に一時的な不整合を起こし得る。booleanとnullableなフィールドでは、
現在地を持たない展開済みツリーなどの不正状態を作れてしまう。データだけをモデル化しても、コマンドと
完了条件がViewに暗黙化したままであり、今回の回帰原因を除去できない。

## 決定を見直す契機

AndroidのナビゲーションAPIが同等のコンパイル時保証を持って状態を所有できる場合、新しい操作を全域な
遷移として表現できない場合、または状態型の計測済み複雑性が防止できる回帰と不正状態を上回る場合に、
具体的な状態分解を見直す。
