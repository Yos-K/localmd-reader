# ADR-0018: 表示文言を一つの言語別ビュー型から完全に構成する

状態: 採用

## 決定事項

ビューアに表示するすべての文言は、選択された`ViewerText`のサブタイプから供給する。Pro機能
カタログのような構造化された文言も、`EnglishViewerText`または`JapaneseViewerText`が同一言語の
完全な値として所有する。ローカライズ済みの外枠と、表示文字列を持つ言語非依存部品を組み合わせない。
presentationは一つの`ViewerText`だけを受け取り、別の言語ソースを個別に選択できないようにする。

## 検討した選択肢

- 機能説明を英語定数としてドメインカタログに残す。
- ダイアログ描画時に文字列ごとの条件分岐で翻訳する。
- AndroidのリソースIDをドメインモデル全体で使う。
- 各言語別ビュー型に、構造化された表示文言一式を所有させる。

## 選択理由

日本語を選択するとは、表示されるすべての文言が日本語になることである。各`ViewerText`サブタイプに
完全なカタログの実装を必須にすると、言語実装の欠落をコンパイル時に検出でき、言語選択を一つの
構成境界へ集約できる。

## 選択しなかった理由

英語のドメイン定数は言語が混在した画面を作る。文字列ごとの条件分岐は言語判断を分散させ、変更漏れを
許す。AndroidのリソースIDはプラットフォーム非依存モデルをAndroidへ結合する。言語別ビュー型なら、
機能の同一性をドメインに残しながら表示文言の供給元を一つにできる。

## 決定を見直す契機

対応言語が増えてカタログ生成が必要になった場合、Androidリソースでも同等の完全性を保証できる場合、
または外部翻訳パイプラインが構造化文言の全項目をビルド時に検証できる場合に見直す。
