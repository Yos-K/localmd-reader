# ADR-0016: プラットフォーム非依存の開いている文書タブ所有

状態: 採用

## 決定事項

`OpenDocumentTabSession`を`OpenDocumentTabs`の唯一の可変な所有者とする。
文書を開く、選択する、前後へ移動する、閉じる、描画済み文書を置換する操作は、
このセッションへコマンドとして渡す。Androidのpresentationコードは現在の不変な
`OpenDocumentTabs`スナップショットを参照できるが、直接保持または置換してはならない。

コントロール更新、アクティブ文書の表示、永続化などAndroid側の完了処理は、
`DocumentTabSessionController`または該当するpresentationアダプタに残す。
描画済み文書の置換結果は小さなハンドラへ通知し、Androidの副作用を
プラットフォーム非依存の所有者へ持ち込まない。

## 検討した選択肢

- `OpenDocumentTabs`を`MainActivity`の可変フィールドとして残す。
- 各文書入力元や移動操作が不変スナップショットを個別に置換する。
- タブと描画の所有権を直ちに一つのworkspace集約へ統合する。
- 開いている文書タブ専用のプラットフォーム非依存な所有者を導入する。

## 選択理由

タブは、ファイル選択、外部Intent、クリップボード下書き、タブ操作、ジェスチャー、
保存、非同期描画完了から変更される。スナップショットの直接代入では、すべての入口が
同じ所有規則を忘れず守ることに正しさが依存していた。唯一の所有者により、常に有効な
タブ集合を保ち、すべての状態変更を明示しつつ、参照には不変スナップショットを使える。

タブと描画の所有者を分けておくことで、`MainActivity`からの移動を検証可能な小さい単位で
進められる。両者が連動する閉じる処理は引き続き`DocumentTabCloseResult`が表現する。

## 選択しなかった理由

`MainActivity`のフィールドではAndroidがドメイン状態の所有者のままになる。呼び出し元ごとの
スナップショット置換は共通動作を迂回でき、実質的な所有者を複数作る。タブと描画の即時統合は、
集約境界とライフサイクル要件が確立する前に今回のリファクタリング範囲を広げる。

## 決定を見直す契機

複数ウィンドウで一つの文書workspaceを共有するとき、開いているタブをActivityより長く
process-level serviceで保持するとき、またはタブと描画セッションを一つのトランザクション集約に
すべき根拠が得られたときに見直す。
