# ADR-0011: メニュー内セクションを折り畳み状態としてモデル化する

状態: 採用

## 決定事項

ナビゲーションメニュー内の設定と目次を、共通のalways-validな`DisclosureState`のvariantである
`Collapsed`と`Expanded`として表現する。`ExpandableMenuSection`が状態からvisibilityへの変換と、
展開中の内容更新を所有する。到達不能な階層型設定ダイアログ実装を削除し、実際のインライン設定操作を
仕様モデルへ反映する。
`TableOfContentsMenuPanel`は目次行の生成と再配色を所有し、`MainActivity`は現在の見出しの提供と
選択された行の処理を担当する。

## 検討した選択肢

- 展開可能なセクションごとにpresentationのbooleanを保持する。
- Android Viewのvisibilityから展開状態を推測する。
- 古い仕様モデルに合わせ、未使用の階層型設定ダイアログを復活させる。
- 開閉の振る舞いを共有しつつ、設定と目次はハーネス上で別の操作状態として維持する。

## 選択理由

両セクションの遷移は、すべてのトグルが有効な折り畳み状態と展開状態を相互に変換する同じ全域関数である。
共通型によりvisibilityが正しい状態になることを防ぎつつ、ハーネスでは異なるコマンドと内容を維持できる。
到達不能なダイアログを削除することで、モデル、実装、ユーザーが実際にたどる操作を一致させ、死んだコードも
減らせる。

## 選択しなかった理由

独立したbooleanとvisibilityによる推測はADR-0009で扱った失敗を繰り返す。ダイアログの復活は設定操作を
遅くし、採用済みのインラインUXに反する。設定と目次をハーネス上でも一つにすると、それぞれ異なる完了・
回復コマンドが失われる。目次行の描画を`MainActivity`に残すと、モデル化されたセクションが無関係な
ビュー生成・配色メソッドへ分散したままになる。

## 決定を見直す契機

セクションが折り畳み・展開以外の状態を必要とする場合、複数セクションを排他的に開く必要が生じた場合、
または設定を専用画面へ移し、そのナビゲーションライフサイクルがインライン開閉を置き換える場合に見直す。
