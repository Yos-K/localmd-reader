# ADR運用

## 目的

ユーザーの指摘、セキュリティ制約、ビルド上の根拠から決めた構造を、後続の機能作業が黙って反転することを防ぐ。

## 手順

1. 機能追加・振る舞い修正の前に `docs/adr/README.ja.md` を読む。
2. 関連する決定と見直し契機を確認する。
3. 指摘により構造を変える場合、先にADRを新規作成、更新、または後継ADRで置換する。
4. テストを先に書き、ルールドキュメントを同期して実装する。
5. すべての `feat:` / `fix:` コミットへ `ADR-Review:` トレーラを1つ付ける。
6. PR作成前に `pr-preflight` を通して `check-adr-review.sh` を実行する。

## ゲート

`check-adr-review.sh` はbaseからHEADまでの機能追加・修正コミットを個別に検査する。コミットは正典ADRを
1件以上参照するか、理由付きの `none` を宣言する。参照ADRはそのコミット時点に存在し、日本語版を持ち、
全ADRは必須5節を備えなければならない。

必須CI `fitness` で実行するため、未宣言は警告ではなくマージ停止となる。

## 限界

ガードが証明するのは、確認宣言とADR形式である。`none` の理由が誠実か、実装がADRの意味に従うかは
機械的には証明できないため、コードレビュー、操作モデル、テストでも確認する。`none` の濫用が見つかった場合は、
対象変更ではADR参照を必須にするなど形式を狭める契機とする。

## 過去履歴の監査

現在の `main` とリポジトリ再作成前のバックアップ履歴を監査し、オフライン境界、Free/Pro境界、
継続的なMarkdownライブラリ、SAF、分離Mermaid描画、Gradle CIの6決定を復元した。
価格、文言、局所的な見映えは、構造移行を伴わず変更されるためADRではなく製品方針・通常仕様で扱う。
